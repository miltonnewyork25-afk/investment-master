# 市场注意力雷达 (Market Attention Radar) v2.0

> Phase 0.5 详细执行规范。由 `docs/deep_dive_protocol.md` v4.0 引用。
> 与 `docs/market_debate_scanner.md` 互补：Debate Scanner 侧重多空争论提炼，Attention Radar 侧重全维度注意力发现与框架覆盖度闭环。

---

## 设计哲学

**问题**: 静态框架假设我们事先知道该分析什么。但市场的焦点每月都在变——上季度关心毛利率，这季度可能关心监管风险。仅依赖预设模块，分析师会遗漏市场当前最在意的维度，导致报告"正确但无人关心"。

**解决方案**: 在分析开始之前，先用 5 路并行搜索发现市场此刻真正关注的维度，再与标准框架逐一比对，为未覆盖的高热维度自动生成 Hot-Patch 模块。

**核心理念**: "让市场告诉我们该分析什么，然后与框架对比找出盲区。"

---

## 触发条件

- **强制触发**: 所有 Tier 3 (`/deep-dive`) 分析
- **推荐触发**: Tier 2 (`/standard-analysis`) 分析（可简化执行，见末尾 Tier 2 简化版）
- **执行时机**: 与 Phase 0 数据预取并行，不额外占用会话
- **前置依赖**: 无（独立于 `data-prefetch`，仅需股票代码和行业分类）

## 执行流程（8步，含3个v2.0新增步骤）

### Step 1: 5路并行WebSearch（过去30天）

启动 5 个并行 WebSearch Agent，分别聚焦不同信息维度：

```yaml
Agent A — 分析师焦点:
  查询: "[TICKER] analyst report [YEAR] key concerns"
  目标: 提取卖方分析师当前最关注的议题
  输出: 分析师聚焦维度列表

Agent B — 多空分歧:
  查询: "[TICKER] bull bear debate [YEAR]"
  目标: 提取投资者之间的核心分歧维度
  输出: 多空争议维度列表

Agent C — 风险与催化剂:
  查询: "[TICKER] risk catalyst upcoming [YEAR]"
  目标: 提取近期风险事件和潜在催化剂
  输出: 事件驱动维度列表

Agent D — 散户情绪:
  查询: "[TICKER] investor sentiment reddit twitter [YEAR]"
  目标: 提取零售投资者的热门话题和情绪倾向
  输出: 散户关注维度列表

Agent E — 行业趋势:
  查询: "[INDUSTRY] disruption trend [YEAR]"
  目标: 提取行业层面的结构性变化
  输出: 行业变迁维度列表
```

**每个 Agent 返回结构化数据**:

```yaml
- dimension: "维度名称（一句话描述）"
  source_count: 3          # 独立信息源数量
  sentiment: "分歧"        # 一致看多 / 一致看空 / 分歧 / 不确定
  recency: "2026-01-28"    # 最近讨论日期
  representative_source: "来源名称+链接"
```

---

### Step 2: Top 10维度抽取与排序

**2.1 合并与去重**

将 5 个 Agent 的结果合并，对语义相似的话题进行聚类去重。判定标准：两个话题如果讨论的底层变量相同（如"FSD进展"和"自动驾驶监管"都指向自动驾驶业务），合并为一个维度并保留更具体的表述。

**2.2 计算注意力分**

```
注意力分 = mention_frequency × source_authority_weight
```

来源权威性权重 (`source_authority_weight`):

| 来源类型 | 权重 | 示例 |
|---------|:----:|------|
| 卖方分析师报告 | 3 | Morgan Stanley、Goldman Sachs 研报 |
| 财经专业媒体 | 2 | Bloomberg、Reuters、WSJ、财新 |
| 社交媒体/论坛 | 1 | Reddit、Twitter/X、雪球 |

注意力分归一化为 0-100 分制（最高分 = 100，其余按比例缩放）。

**2.3 排序与输出**

取排名前 10 的维度，对每个维度提炼核心分歧/争议点。

**输出模板**:

```markdown
| 排名 | 维度 | 注意力分 (0-100) | 核心争议 | 主要来源 |
|:---:|------|:---:|---------|---------|
| 1 | [维度描述] | 95 | [多方认为X，空方认为Y] | [2-3个来源] |
| 2 | [维度描述] | 88 | [核心分歧点] | [2-3个来源] |
| ... | ... | ... | ... | ... |
| 10 | [维度描述] | 42 | [核心分歧点] | [2-3个来源] |
```

---

### Step 2.5: Core Questions 提取（v2.0新增）

> 将 Top 10 维度转化为 5-8 个投资者视角的核心问题。详见 `docs/core_questions_methodology.md`

**2.5.1 维度→问题转化**

对 Top 10 中注意力分≥50的维度，转化为具体的投资问题（Core Question）：

```
维度 → CQ 转化规则:
  1. 维度描述改写为疑问句
  2. 聚焦投资决策影响（不是学术问题）
  3. 语义重叠的维度合并为同一CQ
  4. 最终保留 5-8 个 CQ
```

**2.5.2 CQ质量检验**

每个CQ必须通过三项检验：具体性（指向具体业务变量）、争议性（市场存在分歧）、可回答性（可用数据回答）。

**2.5.3 输出模板**

```markdown
## Core Questions (争议驱动)

| # | Core Question | 来源维度 | 注意力分 | 多方立场 | 空方立场 |
|---|-------------|---------|:---:|---------|---------|
| CQ1 | [问题] | 维度#1,#3 | 95 | [立场] | [立场] |
| CQ2 | [问题] | 维度#2 | 88 | [立场] | [立场] |
| ... | ... | ... | ... | ... | ... |
```

### Step 3: 覆盖度比对 (Coverage Mapping)

将 Top 10 维度逐一与标准框架模块进行映射：

**通用模块池**: M01-M14（公司画像、产业链、财务分析、估值、护城河等）

**行业模块池**（按公司行业选择对应池）:
- 科技平台: TP01-TP06
- 半导体: S01-S06
- 消费品: C01-C06
- 金融: F01-F06

**映射判定规则**:

```yaml
对每个 Top 10 维度:

  如果该维度的核心问题可由某个标准模块自然回答:
    status: "已覆盖"
    action: 在该模块中给予额外篇幅重点分析，标注"市场高关注"
    mapped_module: "M07/S03"（具体模块编号）

  如果该维度的核心问题无法由任何标准模块回答:
    status: "需补丁"
    action: 进入 Step 4 生成 Hot-Patch Module
    reason: "现有框架无对应分析维度"

  如果部分覆盖（标准模块涉及但深度不足）:
    status: "部分覆盖"
    action: 对该标准模块追加分析要求 + 考虑是否需要补丁
    mapped_module: "M05"（标注需加深部分）
```

**覆盖率计算**:

```
覆盖率 = (已覆盖 + 部分覆盖) / 10 × 100%
目标: ≥80%
```

如覆盖率 < 80%，必须通过 Step 4 生成足够的 Hot-Patch 模块使覆盖率达标。

---

### Step 3.5: CQ-模块相关性评分（v2.0新增）

> 为每个模块分配执行权重，让资源集中在核心问题上。详见 `docs/core_questions_methodology.md`

**3.5.1 评分矩阵**

对每个 CQ，评估所有模块的相关性（0-3分）：

| 评分 | 含义 | 模块执行深度 |
|:---:|------|---------|
| 3 | 高相关: 回答CQ的核心素材 | 完整深度 (≥2,000字符) |
| 2 | 中相关: 有用的辅助信息 | 聚焦段落 (500-1,000字符) |
| 1 | 低相关: 关系微弱 | 3行摘要 (≤150字符) |
| 0 | 无关 | 可跳过（仅标题） |

**3.5.2 模块权重确定**

取每个模块在所有CQ中的最高评分作为该模块的执行权重。

**3.5.3 输出**: CQ×模块相关性矩阵（嵌入执行清单）

---

### Step 4: Hot-Patch Module 生成

对每个状态为"需补丁"且注意力分 >= 70（即 heat >= 7）的维度，生成 Hot-Patch 模块规格：

```yaml
Hot-Patch 模块规格模板:

  module_id: "HP-01"                    # 编号: HP-01, HP-02, HP-03...
  title: "Musk-DOGE品牌极化影响评估"      # 模块标题
  source_dimension: "排名#1, 注意力分95"  # 来源维度
  analysis_scope: |                      # 分析范围（2-3句话）
    量化Musk参与DOGE对Tesla品牌资产的正负影响。
    分析主要市场(美国/欧洲/中国)的品牌感知差异。
    评估品牌极化对订单量和定价权的实际影响。
  assigned_phase: "Phase 1"              # 分配到哪个Phase执行
  word_count_target: 2500                # 字数目标: 1,500-3,000字符
  data_requirements:                     # 数据需求
    - "WebSearch: Tesla brand sentiment survey 2026"
    - "WebSearch: Tesla order cancellation rate by market 2026"
    - "WebSearch: Musk DOGE approval rating impact"
  output_spec:                           # 产出要求
    tables: ["品牌感知变化矩阵（按市场）"]
    conclusion: "品牌极化对12个月营收的估算影响（+/-百分比）"
  expiry: "本次报告专用"                   # Hot-Patch为临时模块
```

**Hot-Patch Phase 分配原则**:

| 维度类型 | 分配Phase | 示例 |
|---------|:---:|------|
| 公司身份/品牌/战略方向 | Phase 1 | 品牌极化、战略转型争论 |
| 财务/估值/资本结构 | Phase 2 | 新业务亏损拖累、分拆估值 |
| 竞争/技术/护城河 | Phase 3 | 技术路线之争、新进入者威胁 |
| 风险/黑天鹅/极端情景 | Phase 4 | 监管突变、地缘风险 |
| 时机/仓位/操作 | Phase 5 | 事件驱动交易窗口 |

**关键约束**: Hot-Patch 模块为临时性模块——仅存在于本次报告中，不永久纳入标准框架。如果某类 Hot-Patch 在连续 3 次分析中反复出现，应考虑将其升级为标准模块。


### Step 5.5: CQ驱动报告结构生成（v2.0新增）

> 报告按 Core Questions 分章，而非按模块编号分章。详见 `docs/core_questions_methodology.md`

**5.5.1 章节组织**

```markdown
报告结构:
  执行摘要 (500字)
  CQ1: [最高注意力分的问题] → 引用高相关模块的分析
  CQ2: [第二高的问题] → 引用高相关模块的分析
  ...
  CQ5-8: [剩余问题，可合并]
  综合判断: 估值 + Kill Switch + 行动建议
  附录: 完整数据表格 + 低相关模块摘要
```

**5.5.2 模块内容分配**

一个模块可能服务于多个CQ——将其分析拆分到各相关章节。附录中保留完整版本。

**5.5.3 输出**: CQ驱动的报告大纲（嵌入执行清单）

## 质量门控

Phase 0.5 必须满足以下全部条件方可标记完成：

| 门控项 | 标准 | 不达标处理 |
|-------|------|----------|
| Top 10 维度提取 | >= 10 个维度完成排序 | 追加搜索查询，扩展到行业/宏观层面 |
| 覆盖率 | >= 80% | 生成 Hot-Patch 直到覆盖率达标 |
| 每维度来源数 | >= 2 条独立来源 | 来源不足的维度降级或剔除，补入新维度 |
| Hot-Patch 完整度 | 每个补丁具备 analysis_scope + data_requirements + output_spec | 规格不完整的补丁不计入覆盖率 |
| 执行清单输出 | Phase 1-5 的合并执行计划已生成 | 必须完成后才能启动 Phase 1 |

## 失败模式与防护

| 失败模式 | 症状 | 防护措施 |
|---------|------|---------|
| 搜索结果全是噪音 | 提取不出有意义的维度 | 提高 source_authority_weight 门槛，过滤 social_media 结果；追加行业特定查询 |
| 维度太宏观 | 出现"全球经济衰退"、"利率走向"等泛化话题 | 要求维度必须 company-specific——能直接对应该公司的某个业务/财务/战略变量 |
| 维度与标准模块映射模糊 | 难以判定"已覆盖"还是"需补丁" | 宁可标记"需补丁"也不强行映射。误判为已覆盖的代价（遗漏分析）远大于多写一个补丁的代价 |
| Hot-Patch 数量过多 | 生成 5+ 个补丁，报告篇幅膨胀 | 硬性上限: 单次报告最多 4 个 Hot-Patch；超出时按注意力分取前 4 |
| 搜索时效性不足 | 返回的结果集中在 6 个月前 | 强制限定搜索范围为近 30 天；recency < 30天的结果权重 x0.5 |

| 步骤 | Tier 3 完整版 | Tier 2 简化版 |
