## 评分维度标准 (10维度)

> 来源: GOOGL Phase 5 验证基准。所有后续报告必须使用≥8个维度。

| # | 维度 | 权重 | 说明 |
|:---:|------|:---:|------|
| 1 | 估值吸引力 | 15% | SOTP/DCF收敛度+安全边际 |
| 2 | 增长质量 | 15% | 收入增长结构+利润转化质量 |
| 3 | 护城河强度 | 12% | 五维度护城河评估+侵蚀信号 |
| 4 | 财务健康 | 10% | 当前+前瞻性健康度(双轨) |
| 5 | 管理层质量 | 8% | 领导力记录+内部人交易信号 |
| 6 | 催化剂明确性 | 10% | 近期+中期催化剂清晰度 |
| 7 | 风险可控性 | 10% | 风险集中度+缓解可行性 |
| 8 | 聪明钱信号 | 8% | 机构/内部人/做空综合信号 |
| 9 | 竞争定位 | 7% | 市场定位 vs 同业(Mag7语境) |
| 10 | 时机因素 | 5% | 估值周期+情绪阶段+催化剂时机 |

**评分规则**: 偶数(4/6/8)=确信判断, 奇数(5/7)=边界条件, 全部锚定Phase 4已验证数据

## Kill Switch详细格式标准 (10字段)

> 每个KS必须包含以下10个字段，缺少任何字段=不合格。

| 字段 | 格式要求 | 示例 |
|------|---------|------|
| **触发条件** | 具体决策规则 | "法院命令Chrome结构性剥离" |
| **具体阈值** | 量化边界 | "命令文本含'structural remedy'" |
| **当前状态** | 当前读数+数据源 | "DOJ上诉已提交 [硬数据: DOJ filing]" |
| **当前距离** | 距阈值距离 | "12-18月待裁决" |
| **动作** | 明确组合动作 | "立即清仓; 估值$132/股(-60%)" |
| **CQ关联** | 关联核心问题 | "CQ2(DOJ影响)" |
| **Bear#关联** | 关联Phase 4看空 | "Bear 1 (10%概率)" |
| **数据源** | 验证数据源 | "PACER, SCOTUSblog, Reuters" |
| **AI相关** | AI暴露标记 | Yes/No |
| **紧迫性** | 状态颜色 | 红=迫近 / 黄=监控 / 绿=远期 |

## 可验证预测3情景格式标准

> 每个VP必须包含Base/Bull/Bear三层，禁止单情景预测。

```markdown
### VP-0X: [指标名称] ([验证日期])

| 层级 | 预测值 | 逻辑 |
|:----:|:-----:|------|
| Bear | [数字] | [下行驱动] |
| **Base** | **[数字]** | **[核心假设]** |
| Bull | [数字] | [上行驱动] |

- **验证日期**: YYYY-MM-DD (12个月内)
- **验证数据源**: SEC 10-K / 第三方数据库
- **CQ关联**: CQ#
- **Data Anchor**: [硬数据: source, date]
```

## CQ闭环5要素格式标准

> 每个CQ必须包含以下5要素，缺少任何要素=不合格。

```markdown
## CQ#: [问题名称]

### 最终回答
[1-2段确定性回答，含置信度调整]

### 置信度路径
| Phase | 置信度 | 驱动因素 |
|:---:|:---:|------|
| P0.5 | X% | 初始评估理由 |
| P1/P3/P4 | Y% | Phase特定发现 |
**最终置信度: Z%**

### Kill Switch关联
- 主KS: [映射到具体触发器]
- 监控指标: [量化标志]

### 1年内验证事件
1. [事件1] (具体时间)
2. [事件2] (具体时间)

### "如果我们错了"
[下行场景分析 + 估算影响 $/股]
```

## 仓位建议5档格式标准

| 价格区间 | 推荐仓位 | 条件 | 理由 |
|---------|:---:|------|------|
| >[高估价] | 0% (观望) | — | 高于共识，风险/回报失衡 |
| [高估-公允] | 0-1% (观察仓) | 有明确催化剂 | 仍在调和估值之上 |
| [公允区间] | 2-3% (标准仓位) | 基本面未恶化 | 进入SOTP区间 |
| [低估区间] | 3-5% (核心仓位) | ROI初步验证 | 超过估值下限 |
| <[深度低估] | 5-7% (重仓) | 非系统性风险 | 极端安全边际 |

## Complete报告组装标准

**Phase 5完成后，必须将所有Phase整合为一份Complete文档**:

```
{TICKER}_Complete_v{版本}_{YYYY-MM-DD}.md
```

**组装流程**:
1. 统一TOC（全报告目录，含所有章节号+标题+字符统计）
2. Phase 0.5 Market Radar
3. Phase 1 全文
4. Phase 2 全文
5. Phase 3+3.5 全文
6. Phase 4 全文
7. Phase 5 全文（已按本文档标准升级）
8. 全局免责声明
9. 运行 `tests/quality_gate_complete.sh` 验证
10. 只有通过质量门控后才能 git commit

**禁止**: Phase 5完成即宣布"全量完成"而不组装Complete报告。
