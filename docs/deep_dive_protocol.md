## 启动协议

### Phase 0: 数据基础（自动执行，不占会话）

1. **环境验证** — 确认当前worktree + 公司行业分类
2. **数据预取** — 调用 `data-prefetch` 技能，获取14个数据文件到 `data/research/{TICKER}/`
3. **数据质量检查** — 读取 `prefetch_metadata.json`，确认关键数据可用
4. **预测市场可用性** — 检查 `prediction_market.json` 覆盖度
5. **框架加载** — 读取行业特化配置 + 行业框架文档
6. **Data Master初始化 (v6.0)** — 创建 `reports/{TICKER}/data/shared_context.md` 为DM格式(v1.0)，写入财务/市场基础数据锚点。详见 `docs/data_version_control.md`
7. **KAL初始化 (v6.0)** — 创建 `reports/{TICKER}/data/key_assumptions.md` 空模板。详见 `docs/key_assumptions_list.md`

**Phase 0 完成标准**: `prefetch_metadata.json` 存在 + Layer 1数据全部OK + ≥8/14文件可用 + DM v1.0已创建 + KAL模板已创建

### Phase 0.5: 市场注意力雷达 + Core Questions（自动执行，与Phase 0并行）

> **v4.0新增，v5.0升级**。让市场告诉我们该关注什么，再围绕核心争议组织报告。详见 `docs/market_attention_radar.md` + `docs/core_questions_methodology.md`

1. **5路并行WebSearch** — 30天内分析师报告/多空争论/风险催化/散户情绪/行业趋势
2. **Top 10维度抽取** — 按(提及频率 × 来源权威性)排序市场当前关注点
3. **Core Questions提取** — Top 10 → 5-8个投资者核心问题 (v5.0新增)
4. **覆盖度比对** — 将Top 10与框架预设模块(M01-M14 + 行业模块)逐一匹配
5. **CQ-模块相关性评分** — 每个模块按CQ相关性获得执行权重 (v5.0新增)

8. **输出执行清单** — 标准模块 + Hot-Patch模块合并，输出Phase 1-5完整执行清单

## Fast Gate 强制门控 (v5.1新增)

> **来源**: AMD v3.1 验证——研报完整版 90,413字符全部7门控通过，证明 research_fast.sh 可作为可靠质量关卡。

**铁律**: 每个Phase完成后，**必须**运行 Fast Gate 检查，通过后写入checkpoint，再 git commit。

```bash
# Phase完成后执行（在对应行业worktree中）:
bash tests/research_fast.sh reports/{TICKER}/{TICKER}_Phase{N}_v{版本}_{YYYY-MM-DD}.md {最低字符数} 3
# 通过(exit 0) → 写入/更新 checkpoint.yaml → git commit
# 失败(exit 1) → 修复后重新检查
```

**Checkpoint写入**: Fast Gate通过后、git commit前，写入/更新 `reports/{TICKER}/data/checkpoint.yaml`。详见 `docs/checkpoint_protocol.md`

**7项门控**:
| 门控 | 检查内容 | 阈值 |
|------|---------|------|
| G1 | 字符数(wc -m) | ≥Phase目标字符数 |
| G2 | 数据标注密度 | ≥15/万字符 (Phase 4: ≥8/万字符) |
| G3 | 免责声明+目录 | 必须存在 |
| G4 | Mermaid图表 | Tier 3建议≥3个 |
| G5 | 禁止词检查 | 0个违规 |
| G6 | 标注格式校验 | 无超长未闭合标注 |
| G7 | Kill Switch/看空 | Phase 4: KS≥15提及, 看空≥8 |

**Phase 4 标注密度例外** (v5.1): Phase 4以分析判断为主（行为金融偏差检查、事实核查、反证挑战），数据标注密度天然低于Phase 1-3。G2阈值对Phase 4单独设为 **8/万字符**（非15/万）。AMD Phase 4实测密度7.0，属预期行为。

**标注格式双兼容**: 旧格式`[A:]`/`[B:]`/`[P:]`/`[E:]`和新格式`[硬数据:]`/`[合理推断:]`/`[主观判断:]`均被 Fast Gate 识别。AMD v3.1 用旧格式达到16.3/万字符密度。

### 双维度质量门控 (v6.0新增)

> Fast Gate 升级为 P-G（过程门控）的自动化基础，新增 R-G（结果门控）检查数据一致性。详见 `docs/quality_gate_v2.md`

**P-G (过程门控)**: 10项 — 继承Fast Gate 7项 + CQ覆盖 + 标注比例 + 看空篇幅
**R-G (结果门控)**: 12项 — SOTP一致性 + KS一致性 + KAL一致性 + DM一致性 + 估值可追溯 + 数字一致性

**通过标准**: P-G ≥ 8/10 **且** R-G ≥ 7/10
**Phase 4/5强制**: 必须通过全部12项R-G检查

```bash
# Phase完成后执行:
# Step 1: 自动P-G (Fast Gate)
bash tests/research_fast.sh reports/{TICKER}/{file} {min_chars} 3
# Step 2: Agent R-G检查 (Phase 4/5用模板)
# 详见 docs/quality_gate_v2.md R-G检查执行模板
```

**必做模块**:
- 公司基本画像（业务模型、历史沿革、管理层评估）
- 产业链上下游映射（供应商、客户、竞争者生态）
- 预测市场环境扫描（宏观+行业+公司特定事件概率）

- **M14呈现**: 展示Phase 0.5的Top 10维度+CQ清单+覆盖计划+Hot-Patch分配

**字数目标**: ≥20,000字符 (wc -m)

### Phase 3.5: AI深度评估（v4.0新增，强制执行）

> **设计原因**: AI浪潮下，公司级"AI受益者/被颠覆者"标签掩盖了分部间的巨大差异。同一公司的不同业务对AI的暴露度可能截然相反。详见 `docs/ai_depth_assessment.md`

**产出**: 分部级AI冲击矩阵 + AI实施深度L×S定位 + AI调整后估值

**必做三层分析**:

**Layer 1 — 分部级AI冲击矩阵 (M13)**
- 对每个主要业务分部(占营收≥90%)独立评估5个维度:
  - 收入冲击(-5~+5): AI对该分部收入的增减影响
  - 成本冲击(-5~+5): AI对该分部成本结构的改变
  - 护城河变化(强化/中性/削弱): AI对现有竞争优势的影响
  - 竞争格局变化(利好/中性/利空): AI是否改变竞争者进入壁垒
  - 时间窗口(1-3yr/3-5yr/5-10yr): 冲击兑现的时间范围
- 每个分部归类为: AI放大器 / AI赋能者 / AI中性 / AI易受冲击 / AI被颠覆
- 计算: 概率加权AI净分 = Σ(分部AI净分 × 收入权重 × 实现概率)

**Layer 2 — AI实施深度评级**
- 引用 `skills/ai_assessment/` 框架:
  - L轴(实施级别): L0观察 → L1决策支持 → L2受控自动化 → L3自主运营 → L4完全自主
  - S轴(商业兑现): S0叙事期权 → S1早期变现 → S2规模化 → S3成熟 → S4平台化
  - 五不变量检验: 区分AI叙事噪音 vs 真实进展
- 输出: L×S坐标定位 + 五不变量通过率 + 同业对比

**Layer 3 — AI估值影响量化**
- 从Layer 1+2推导AI对Phase 2 SOTP估值的调整:
  - AI溢价分部: 估值乘数上调幅度及依据
  - AI折价分部: 估值乘数下调幅度及依据
  - AI期权分部: 概率加权期权价值
- 输出: "AI调整后SOTP" vs "基线SOTP" = AI隐含溢价/折价

**字数目标**: ≥10,000字符 (wc -m)

**门控 QG-09.5**:
- 所有主要分部(占营收≥90%)已完成AI冲击评分
- L×S坐标已定位且有≥3条证据支撑
- AI调整估值与基线估值差异已量化并解释

### Phase 4: 对抗审查（v4.0升级为强制执行，v5.0增加看空等权重）

**产出**: 行为金融偏差修正 + 事实核查 + 反证挑战 + Smart Money验证 + **看空等权重分析**

> **v4.0强化**: 从v3.0的"建议执行"升级为**强制执行**，禁止跳过。
> **v5.0新增**: 看空分析等权重方法论，详见 `docs/bear_case_methodology.md`

**必做模块**:
- **行为金融四项偏差检查**（详见 `docs/behavioral_finance.md`）
  - 锚定效应: 目标价是否被当前股价锚定？
  - 确认偏误: 是否选择性忽略反面证据？
  - 过度自信: 概率区间是否过窄？
  - 损失厌恶: 是否因沉没成本维持已不成立的论点？
- **关键数据事实核查**: 抽查≥10个核心数据点的来源真实性
- **反证挑战**: "如果论点完全错误，最可能的原因是什么？" × 3条
- **看空等权重分析 (v5.0新增)**: ≥8个看空论点，每个含触发条件+概率+影响量化+时间窗口。执行钢人论证。详见 `docs/bear_case_methodology.md`
- **Smart Money立场验证**: 机构持仓变化 vs 我们的结论是否一致
- **极端压力测试**: SOTP极端场景 + AI极端场景
- **维度回检**: Phase 0.5的Top 10维度是否全部在报告中得到回应
- **"So What?"抽查 (v5.0新增)**: 随机抽查5个模块，验证洞察密度。详见 `docs/differentiated_insight_standard.md`

**看空篇幅要求 (v5.0)**: 看空相关内容（Bear Case+风险+反证+Kill Switch+压力测试+竞争威胁）≥总篇幅30%

**字数目标**: ≥15,000字符 (wc -m)

**门控 QG-10~11**:
- QG-10: 四项认知偏差检查全部完成 + 量化修正
- QG-11: 事实核查≥10个数据点 + 反证≥3条 + Top 10维度回应率100%

### Phase 5: 决策输出 (v7.0升级: 质量基准对齐)

**产出**: 最终评级 + 仓位建议 + Kill Switch + 可验证预测 + 行动清单 + CQ最终解答 + **Complete报告**

> **v7.0变化**: Phase 5字符目标从15K提升至34K(GOOGL实际73.6K)。评分从自定维度升级为标准10维度。VP从单情景升级为三情景(Base/Bull/Bear)。KS从简要格式升级为10字段详细格式。新增CQ最终解答(5要素)。Complete报告组装为强制步骤。详见 `docs/quality_benchmarks.md`

**必做模块**:

**1. 综合评分 (10维度加权, 0-100分)**
- **必须使用≥8个评分维度**（标准10维度见 `docs/quality_benchmarks.md`）:
  - 估值吸引力(15%) | 增长质量(15%) | 护城河强度(12%) | 财务健康(10%)
  - 管理层质量(8%) | 催化剂明确性(10%) | 风险可控性(10%) | 聪明钱信号(8%)
  - 竞争定位(7%) | 时机因素(5%)
- 评分规则: 偶数=确信判断, 奇数=边界条件, 全部锚定Phase 4已验证数据
- 评级: 强烈推荐(≥80) / 推荐(65-79) / 中性关注(55-64) / 回避(<55)

**2. 仓位建议 (5档矩阵)**
- 基础仓位 × 周期系数 × 置信度 × AI调整系数
- 必须包含5个价格区间档位（观望/观察仓/标准/核心/重仓）
- 每档含: 仓位% + 条件 + 理由
- 建仓策略: 分批建仓(≥3批) + 时间间隔

**3. Kill Switch注册表 (≥15个, 10字段详细格式)**
- 每个KS必须包含10字段: 触发条件 + 具体阈值 + 当前状态 + 当前距离 + 动作 + CQ关联 + Bear#关联 + 数据源 + AI相关 + 紧迫性
- 含≥2个AI相关KS (v5.1)
- 仪表盘矩阵: 红/黄/绿分布统计

**4. 可验证预测清单 (≥20个, 三情景格式)**
- **每个预测必须包含Base/Bull/Bear三层** (禁止单情景预测)
- 每层含: 预测值 + 逻辑
- 附加: 验证日期(12个月内) + 验证数据源 + CQ关联 + Data Anchor
- 格式示例见 `docs/quality_benchmarks.md`

**5. CQ最终解答 (5要素闭环, v7.0新增)**
- 对每个Core Question输出最终回答:
  - 最终回答(1-2段确定性回答)
  - 置信度路径(各Phase置信度变化表)
  - Kill Switch关联(映射到具体触发器)
  - 1年内验证事件(≥2个具体事件+时间)
  - "如果我们错了"(下行场景+$/股影响)

**6. 投资日历 (12个月滚动)**
- 未来12个月关键事件 + 影响方向 + CQ影响 + 建议行动 + P0-P3优先级
- 含季度检查清单

**7. 90天行动清单**
- 4阶段月级颗粒度 + 具体买入/加仓/减仓触发点
- 含Mermaid决策树(盈利后情景→组合动作)

**8. 最终SOTP估值 (多方法收敛)**
- Phase 4修正后SOTP + DCF + 可比 + 概率加权 → 多方法收敛
- 置信区间: 50%/80%/95%
- 偏差<3%视为"高质量"收敛

**字数目标**: ≥34,000字符 (wc -m) — GOOGL实际73,584字符

**门控 QG-12 (v7.0升级)**:
- Kill Switch≥15个条件（含≥2个AI相关, 10字段详细格式）
- 可验证预测≥20个（三情景格式, ≥80%含Base/Bull/Bear）

**Engine 3: Smart Money Tracker** — 顶级投资者逻辑解码+机构共识度量化

### 五引擎数据依赖矩阵 (v5.0新增)

> 当两个引擎共享数据源时，结论一致不算"双重验证"。

| 引擎 | 独立数据源 | 与其他引擎的数据重叠 |
|------|-----------|-------------------|
| Engine 1 (周期) | 宏观指标、行业库存/产能数据 | 与E4部分重叠(宏观信号) |
| Engine 2 (股权) | 13F持仓、内部人交易、机构进出 | 与E3高度重叠(30-50%) |
| Engine 3 (聪明钱) | 对冲基金持仓、期权异动 | 与E2高度重叠(30-50%) |
| Engine 4 (信号) | 技术指标、资金流、新闻情绪 | 与E1/E5部分重叠 |
| Engine 5 (预测市场) | Polymarket/Kalshi概率数据 | 基本独立 |

**验证规则**:
- E2+E3 结论一致 → 计为1票（非2票），因数据重叠
- E1+E5 结论一致 → 计为2票，数据源独立
- 最终置信度 = 独立数据源支持的引擎数 / 总引擎数

## 铁律（Deep-Dive强制，十五条）

1. **数据必有源**: 每个关键数字标注来源，≥15标注/万字
2. **判断必有据**: 每个核心判断≥2条证据链
3. **预测必可验**: 概率+时间+触发条件
4. **洞察必反证**: "但如果___则不成立"
5. **结论必可行**: 买卖建议+仓位+时间+Kill Switch(≥15个)
6. **概率必背离分析**: 重要风险事件执行PPDA分析
7. **情绪必市场验证**: 投资情绪判断必须用PMSI等指标验证
8. **AI必分部评估**: 禁止仅给公司级AI标签，必须逐分部评分(M13)
9. **热点必回应**: Phase 0.5发现的Top 10维度，报告中必须100%回应(M14)
10. **标注必三层** (v5.0): 每个数据点归类为硬数据/合理推断/主观判断，禁用旧置信度格式
11. **看空必等权** (v5.0): 看空相关内容≥总篇幅30%，≥8个看空论点含四要素
12. **模块必So What** (v5.0): 每个模块通过"So What?"检验，无洞察的模块压缩
13. **Phase必Fast Gate** (v5.1): 每Phase完成→运行research_fast.sh→通过后才能git commit
14. **无源数字禁写入** (v6.0): 报告中每个数字必须来自DM锚点/外部来源/明确公式，无源数字=幻觉=禁止。详见 `docs/anti_hallucination_protocol.md`
15. **估值修正必审计** (v6.0): 每次估值调整走修正管道(L1-L5)，单层≤±15%，记审计日志。详见 `docs/valuation_correction_hierarchy.md`

## 并行Agent加速

> 并行基础见 `docs/parallel_execution.md`
> 协作协议见 `docs/agent_collaboration_protocol.md`

独立任务**必须并行**执行。推荐并行组合:
- Phase 0 + 0.5: 数据预取与注意力雷达并行
- Phase 3各子模块: 护城河/五引擎/热点补丁可并行
- Phase 3.5: Layer 1(冲击矩阵) + Layer 2(L×S定位)可并行
- Phase 5: Kill Switch + 预测清单 + 行动清单可并行

### 多Agent协作机制 (v1.1, AMD验证)

**每个Phase并行执行时必须遵循**:

1. **共享上下文**: dispatch前编写 `reports/{TICKER}/data/shared_context.md` (Data Master格式)，Agent只读此文件
2. **任务锁**: dispatch前创建 `reports/{TICKER}/data/current_tasks/Agent{X}.lock.md`，完成+QG通过后删锁
3. **质量门控**: Agent返回后运行 P-G (Fast Gate) + R-G (结果检查)。详见 `docs/quality_gate_v2.md`
4. **增量提交**: 每个Agent通过QG → 立即 `git commit`
5. **状态仪表盘**: 每个Agent完成时更新 `STATUS.md`
6. **Session恢复**: 新session检测到锁文件 → 执行恢复协议（检查output→判断状态→报告用户）
7. **反幻觉注入 (v6.0)**: 每个SubAgent prompt必须包含反幻觉5条禁令。详见 `docs/anti_hallucination_protocol.md`
8. **DM引用规范 (v6.0)**: Agent引用数据必须附带 `[DM-xxx vN.N]` 版本号。详见 `docs/data_version_control.md`

### 并行Agent Prompt规范 (v5.1+v6.0)

**每个SubAgent prompt必须包含** (v5.1):
1. **具体数据需求列表**: 列出每个数据点/指标（非泛泛描述）
2. **Phase前序发现**: 已完成Phase关键结论摘要
3. **CQ关联**: 与哪个Core Question相关
4. **字符目标**: 期望返回字符数范围
5. **输出格式要求**: 标注格式+表格+Mermaid图需求
6. **验证要求**: 标注数据来源和获取时间
7. **反幻觉5条禁令 (v6.0)**: 详见 `docs/anti_hallucination_protocol.md`
8. **DM/KAL引用指令 (v6.0)**: 详见 `docs/parallel_execution.md` v4.0
