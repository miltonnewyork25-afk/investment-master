### Phase 0: 数据基础 + 温度预筛选（自动执行，不占会话）

1. **环境验证** — 确认当前worktree + 公司行业分类
2. **数据预取升级 (v8.0)** — 调用 `data-prefetch` 技能，获取17个数据文件到 `data/research/{TICKER}/`
   - **原有14类**: 基础财务、市场数据、分析师预期等
   - **新增3类**: 宏观温度数据(CAPE/Buffett/ERP)、预测市场事件、内部人交易情绪
3. **投资温度计算 (v8.0新增)** — 调用 `investment-logic-toolkit` 执行5秒温度评估
   - **宏观温度**: 基于100baggers.club的CAPE/Buffett/ERP数据
   - **基本面质量**: 基于FMP ratios历史数据
   - **市场情绪**: 基于技术指标+内部人交易
   - **输出**: 温度评分(-2.0~+2.0) + 5级分类 + Tier建议
4. **数据质量检查** — 读取 `prefetch_metadata.json`，确认关键数据可用
5. **预测市场可用性** — 检查 `prediction_market.json` 覆盖度
6. **框架加载** — 读取行业特化配置 + 行业框架文档
7. **Data Master初始化 (v6.0)** — 创建 `reports/{TICKER}/data/shared_context.md` 为DM格式(v1.0)，写入财务/市场基础数据锚点。详见 `docs/data_version_control.md`
8. **KAL初始化 (v6.0)** — 创建 `reports/{TICKER}/data/key_assumptions.md` 空模板。详见 `docs/key_assumptions_list.md`
9. **温度路由决策 (v8.0新增)** — 基于温度评分建议最适合的分析深度
   - **≥1.5 (过热)**: 建议Tier 1快速扫描，关注风险
   - **0.5~1.5 (偏热)**: 建议Tier 2标准分析，谨慎评估
   - **-0.5~0.5 (中性)**: 建议Tier 2标准分析
   - **-1.5~-0.5 (偏冷)**: 建议Tier 3深度研究，潜在机会
   - **≤-1.5 (极冷)**: 建议Tier 3深度研究，重点机会

**Phase 0 完成标准**: `prefetch_metadata.json` 存在 + Layer 1数据全部OK + ≥11/17文件可用 + DM v1.0已创建 + KAL模板已创建 + 投资温度已计算

## Fast Gate 强制门控 (v5.1新增)

> **来源**: AMD v3.1 验证——研报完整版 90,413字符全部7门控通过，证明 research_fast.sh 可作为可靠质量关卡。

**铁律**: 每个Phase完成后，**必须**运行 Fast Gate 检查，通过后写入checkpoint，再 git commit。

```bash
# Phase完成后执行（在对应行业worktree中）:
bash tests/research_fast.sh reports/{TICKER}/{TICKER}_Phase{N}_v{版本}_{YYYY-MM-DD}.md {最低字符数} 3
# 通过(exit 0) → 写入/更新 checkpoint.yaml → git commit
# 失败(exit 1) → 修复后重新检查
```

**Checkpoint写入**: Fast Gate通过后、git commit前，写入/更新 `reports/{TICKER}/data/checkpoint.yaml`。详见 `docs/checkpoint_protocol.md`

**7项门控**:
| 门控 | 检查内容 | 阈值 |
|------|---------|------|
| G1 | 字符数(wc -m) | ≥Phase目标字符数 |
| G2 | 数据标注密度 | ≥15/万字符 (Phase 4: ≥8/万字符) |
| G3 | 免责声明+目录 | 必须存在 |
| G4 | Mermaid图表 | Tier 3建议≥3个 |
| G5 | 禁止词检查 | 0个违规 |
| G6 | 标注格式校验 | 无超长未闭合标注 |
| G7 | Kill Switch/看空 | Phase 4: KS≥15提及, 看空≥8 |

**Phase 4 标注密度例外** (v5.1): Phase 4以分析判断为主（行为金融偏差检查、事实核查、反证挑战），数据标注密度天然低于Phase 1-3。G2阈值对Phase 4单独设为 **8/万字符**（非15/万）。AMD Phase 4实测密度7.0，属预期行为。

**标注格式双兼容**: 旧格式`[A:]`/`[B:]`/`[P:]`/`[E:]`和新格式`[硬数据:]`/`[合理推断:]`/`[主观判断:]`均被 Fast Gate 识别。AMD v3.1 用旧格式达到16.3/万字符密度。

### Phase 2: 财务与价格含义

**产出**: 财务数据组织 + Reverse DCF(价格隐含假设) + [深挖]问题深入分析

> **v2.0变化**: 从"给出目标价"转为"解读市场定价隐含了什么假设"。Reverse DCF是核心工具——不是算公司值多少钱，而是反推当前价格在赌什么。传统SOTP/DCF作为参考框架呈现，不作为结论。

**必做模块**:
- **Reverse DCF（核心）+ 承重墙脆弱度表 (v10.0)**: 当前股价隐含了多少增长率/利润率/年限？这些假设合理吗？每个隐含假设用承重墙脆弱度表评估脆弱性。

**承重墙脆弱度表模板**:
```markdown
| 承重墙(隐含假设) | 当前价格隐含值 | 历史/行业参考 | 脆弱度 | 若倒塌影响 |
|-----------------|:------------:|:----------:|:-----:|:--------:|
| 收入增速(5Y CAGR) | 25% | 行业中位15% | 高 | -35% |
| 稳态利润率 | 30% | 当前22%,同行25% | 中 | -20% |
| 增长持续年限 | 15年 | 科技公司均值8年 | 高 | -40% |
| 折现率隐含 | 8% | 当前无风险4.5%+ERP | 低 | -10% |
```
**脆弱度判定**: 高=隐含值超历史/行业参考50%+ | 中=超10-50% | 低=在参考范围内

- 周期定位（行业周期模型，定位当前位置）
- 5年财务趋势分析（收入/利润/现金流/资产负债）— 数据组织，不下估值结论
- 三情景财务推演（Bull/Base/Bear的关键变量范围）
- **资本配置分析**: R&D效率、CapEx ROI、分红/回购策略 — 属于[深挖]区，AI可以深入分析
- **[深挖]问题执行**: Phase 0.5路由到本Phase的深挖类问题在此展开
- **参考框架（[呈现]区）**: SOTP/DCF/可比公司作为多视角参考呈现，标注"参考框架，非目标价"

**字数目标**: ≥25,000字符 (wc -m)

**门控 QG-04~06**:
- QG-04: 周期定位有≥4个支撑信号
- QG-05: Reverse DCF隐含假设已明确列出 + **承重墙脆弱度表完成** + 合理性评估完成
- QG-06: 参考框架≥2种方法呈现（标注为参考，非结论）

### Phase 3.5: AI深度评估（v4.0新增，强制执行）

> **设计原因**: AI浪潮下，公司级"AI受益者/被颠覆者"标签掩盖了分部间的巨大差异。同一公司的不同业务对AI的暴露度可能截然相反。详见 `docs/ai_depth_assessment.md`

**产出**: 分部级AI冲击矩阵 + AI实施深度L×S定位 + AI调整后估值

**必做三层分析**:

**Layer 1 — 分部级AI冲击矩阵 (M13)**
- 对每个主要业务分部(占营收≥90%)独立评估5个维度:
  - 收入冲击(-5~+5): AI对该分部收入的增减影响
  - 成本冲击(-5~+5): AI对该分部成本结构的改变
  - 护城河变化(强化/中性/削弱): AI对现有竞争优势的影响
  - 竞争格局变化(利好/中性/利空): AI是否改变竞争者进入壁垒
  - 时间窗口(1-3yr/3-5yr/5-10yr): 冲击兑现的时间范围
- 每个分部归类为: AI放大器 / AI赋能者 / AI中性 / AI易受冲击 / AI被颠覆
- 计算: 概率加权AI净分 = Σ(分部AI净分 × 收入权重 × 实现概率)

**Layer 2 — AI实施深度评级**
- 引用 `skills/ai_assessment/` 框架:
  - L轴(实施级别): L0观察 → L1决策支持 → L2受控自动化 → L3自主运营 → L4完全自主
  - S轴(商业兑现): S0叙事期权 → S1早期变现 → S2规模化 → S3成熟 → S4平台化
  - 五不变量检验: 区分AI叙事噪音 vs 真实进展
- 输出: L×S坐标定位 + 五不变量通过率 + 同业对比

**Layer 3 — AI对价格含义的影响**
- 从Layer 1+2推导AI如何改变Reverse DCF的隐含假设:
  - AI放大分部: 价格隐含增长率中有多少依赖AI兑现？
  - AI受压分部: 若AI颠覆发生，当前定价是否已反映？
  - AI期权分部: 市场是否为尚未兑现的AI期权付费？
- 输出: "AI调整后隐含假设" vs "基线隐含假设" = AI定价溢价/折价的归因

**字数目标**: ≥10,000字符 (wc -m)

**门控 QG-09.5**:
- 所有主要分部(占营收≥90%)已完成AI冲击评分
- L×S坐标已定位且有≥3条证据支撑
- AI调整估值与基线估值差异已量化并解释

### Phase 4: 对抗审查（v6.0 — 纠错回流机制）

**产出**: 行为金融偏差修正 + 事实核查 + 反证挑战 + Smart Money验证 + **看空等权重分析** + **纠错清单(用于回流)**

> **v6.0核心变化**: Phase 4发现的事实性错误**必须回流覆盖到Phase 1-3**。Complete报告中读者看到的是正确数据，不是"错→纠错"的过程。Phase 4在Complete中只保留**方法论+结构性挑战**，不保留已被覆盖的数据纠正条目。
>
> **原则**: Phase 4的价值在于**对抗思维过程**（认知偏差检测、看空钢人论证、结构性风险），不在于展示"我找到了N个错误"。错误被修正后，读者不需要知道它们曾经存在。

**必做模块**:
- **红队七问 (v10.0新增)**: RT-1~RT-7固定问题逐一回答。详见 `docs/red_team_protocol.md`
  - RT-1 承重墙测试 | RT-2 认知偏差审计 | RT-3 空头钢人 | RT-4 数据质量审计
  - RT-5 黑天鹅压力测试 | RT-6 时间框架挑战 | RT-7 替代解释
- **行为金融偏差检查**（详见 `docs/behavioral_finance.md`，已融入RT-2）
- **关键数据事实核查**: 抽查≥10个核心数据点的来源真实性（已融入RT-4）
- **反证挑战**: "如果论点完全错误，最可能的原因是什么？" × 3条（已融入RT-3）
- **看空等权重分析**: ≥8个看空论点，每个含触发条件+影响量化+时间窗口。执行钢人论证。详见 `docs/bear_case_methodology.md`
- **黑天鹅概率加权表 (v10.0)**: ≥3个黑天鹅事件量化。详见 `docs/bear_case_methodology.md`（已融入RT-5）
- **Smart Money立场验证**: 机构持仓变化 vs 我们的结论是否一致
- **维度回检**: Phase 0.5的Top 10维度是否全部在报告中得到回应
- **"So What?"抽查**: 随机抽查5个模块，验证洞察密度

**Phase 4产出分两类**:

| 类型 | 处理方式 | Complete中呈现 |
|------|---------|--------------|
| **事实性纠错** (数据错误/计算偏差/来源失实) | **回流覆盖**到Phase 1-3原文 | ❌ 不单独呈现 |
| **结构性挑战** (看空论点/认知偏差/压力测试) | 保留为Phase 4核心内容 | ✅ 保留 |
| **CQ置信度调整** | 回流到Phase 5 CQ闭环表 | ❌ 不重复呈现 |

**看空篇幅要求**: 看空相关内容（Bear Case+风险+反证+Kill Switch+压力测试+竞争威胁）≥总篇幅30%

**字数目标**: ≥15,000字符 (wc -m)

**门控 QG-10~11**:
- QG-10: **红队七问(RT-1~RT-7)全部回答**，每个≥200字 + 量化校正
- QG-11: 事实核查≥10个数据点 + 反证≥3条 + Top 10维度回应率100% + **黑天鹅概率加权表≥3个事件**
- **QG-11b (v6.0新增)**: 纠错回流清单已生成，每条标注[回流目标Phase+章节]
- **QG-11c (v9.1新增)**: 回流无痕化 — 回流修正后的Phase 1-3文本中**不得保留任何"P4回流"标注**。修正数据应看起来如同原始内容，来源标注使用原始数据源（如"FMP key-metrics"），不标注"P4回流F03"。读者不应察觉哪些数据曾被修正过。

### 报告结构连贯性规则 (v9.1新增)

**1. 核心论点综合 (深度优先)**
- **一句话结论**: 用一句话概括这家公司的投资命题（不含买卖信号）
- **10维度定性评估**: 估值吸引力 | 增长质量 | 护城河强度 | 财务健康 | 管理层质量 | 催化剂明确性 | 风险可控性 | 聪明钱信号 | 竞争定位 | 时机因素
  - 每个维度: 定性判断(强/中/弱) + 1-2句关键证据 + 置信区间(高/中/低)
  - **不给分数** — "护城河: 强(高置信) — 刻蚀市占率52%+CSBG锁定"比"护城河: 7.2/10"更诚实
  - 评级仍保留四档: 深度关注(强烈看好) / 关注(偏积极) / 中性关注(观望) / 审慎关注(偏谨慎)
- **AI深度加成区**: 在AI有知识深度优势的维度上加倍展开:
  - 技术架构拆解（工艺节点演进/设备世代/物理极限）
  - 供应链交叉验证（客户A数据 vs 供应商B数据 → 一致性检验）
  - 历史周期类比（本轮 vs 2018/2021周期的结构差异）
  - 跨公司财务模式（同行业公司是否呈现相同趋势）

**2. Kill Switch注册表 (10-15个, 9字段格式)**
- 每个KS必须包含9字段: 触发条件 + 具体阈值 + 当前状态 + 当前距离 + 论文含义 + CQ关联 + Bear#关联 + 数据源 + 紧迫性
- "论文含义"描述触发后对投资论文的影响，不包含操作指令
- 含≥2个AI相关KS
- **精选原则**: 每个KS必须有明确的可观测阈值。"宏观环境恶化"不是KS，"10Y国债收益率>5.5%持续3个月"才是KS。

**3. 追踪信号清单 (替代旧VP预测)**
- **5-8个追踪信号** — 不是"我预测X会发生"，而是"投资者应该追踪X指标，因为它能最早反映Y假设是否成立"
- 每个信号包含:
  - **追踪什么**: 具体可观测指标
  - **为什么重要**: 它验证/否定哪个核心假设
  - **当前读数**: 最新数据 + 趋势方向
  - **关键阈值**: 什么水平意味着论文需要修正
  - **数据源**: 在哪里查（SEC/第三方/财报电话会）
  - **CQ关联**: 关联哪个核心问题
- **特异性测试**: 如果信号替换公司名后对同行业任何公司都成立 → 删除

**4. CQ最终解答 (5要素闭环) + CQ置信度演化表 (v10.0)**
- 对每个Core Question输出最终回答:
  - 最终回答(1-2段，明确说出"我们知道什么"和"我们不知道什么")
  - 置信度路径(各Phase置信度变化表)
  - Kill Switch关联(映射到具体触发器)
  - 1年内验证事件(≥2个具体事件+时间)
  - "如果我们错了"(下行场景分析)
- **CQ置信度演化表**: 所有CQ的P0.5→P5置信度变化+加权平均。详见 `docs/core_questions_methodology.md` §4

**5. 价格含义总结 (Reverse DCF — AI最强输出)**
- **这是Phase 5最有价值的部分，应占最大篇幅**
- Reverse DCF隐含假设汇总: 当前价格隐含了多少增长率/利润率/年限？
- 隐含假设合理性检验: 用Phase 1-3的技术/竞争/财务分析检验每个隐含假设
- 参考框架交叉对照: SOTP/DCF/可比公司 — 标注为"参考视角，非结论"
- 条件估值范围: "如果[假设A]成立，合理区间为$X-Y"（非点数目标价）
- 不确定性诚实度: 方法间离散度>30%时标注"高度不确定，多方法未收敛"
- **我们不知道什么**: 明确列出影响估值但无法可靠估计的关键未知

**6. 关键事件日历 (12个月)**
- 已确定的事件(财报/会议/产品发布) + 对哪个CQ/KS有影响

## 铁律（Deep-Dive强制，十五条）

1. **数据必有源**: 每个关键数字可追溯到DM锚点，DM覆盖率≥90%
2. **判断必有据**: 每个核心判断≥2条证据链
3. **预测必可验**: 概率+时间+触发条件
4. **洞察必反证**: "但如果___则不成立"
5. **结论必可追踪**: 核心论点+关键假设+追踪信号+Kill Switch(≥15个)
6. **概率必背离分析**: 重要风险事件执行PPDA分析
7. **情绪必市场验证**: 投资情绪判断必须用PMSI等指标验证
8. **AI必分部评估**: 禁止仅给公司级AI标签，必须逐分部评分(M13)
9. **热点必回应**: Phase 0.5发现的Top 10维度，报告中必须100%回应(M14)
10. **数据必审计** (v10.0): DM锚点三层分类(H/R/S)+脚本验证+文末审计摘要，报告正文零内联标注
11. **看空必等权** (v5.0): 看空相关内容≥总篇幅30%，≥8个看空论点含四要素
12. **模块必So What** (v5.0): 每个模块通过"So What?"检验，无洞察的模块压缩
13. **Phase必Fast Gate** (v5.1): 每Phase完成→运行research_fast.sh→通过后才能git commit
14. **无源数字禁写入** (v6.0): 报告中每个数字必须来自DM锚点/外部来源/明确公式，无源数字=幻觉=禁止。详见 `docs/anti_hallucination_protocol.md`
15. **估值修正必审计** (v6.0): 每次估值调整走修正管道(L1-L5)，单层≤±15%，记审计日志。详见 `docs/valuation_correction_hierarchy.md`

## 并行Agent加速

> 并行基础见 `docs/parallel_execution.md`
> 协作协议见 `docs/agent_collaboration_protocol.md`

独立任务**必须并行**执行。推荐并行组合:
- Phase 0 + 0.5: 数据预取与注意力雷达并行
- Phase 3各子模块: 护城河/五引擎/热点补丁可并行
- Phase 3.5: Layer 1(冲击矩阵) + Layer 2(L×S定位)可并行
- Phase 5: Kill Switch + 预测清单 + 行动清单可并行

### 多Agent协作机制 (v1.1, AMD验证)

**每个Phase并行执行时必须遵循**:

1. **共享上下文**: dispatch前编写 `reports/{TICKER}/data/shared_context.md` (Data Master格式)，Agent只读此文件
2. **任务锁**: dispatch前创建 `reports/{TICKER}/data/current_tasks/Agent{X}.lock.md`，完成+QG通过后删锁
3. **质量门控**: Agent返回后运行 P-G (Fast Gate) + R-G (结果检查)。详见 `docs/quality_gate_v2.md`
4. **增量提交**: 每个Agent通过QG → 立即 `git commit`
5. **状态仪表盘**: 每个Agent完成时更新 `STATUS.md`
6. **Session恢复**: 新session检测到锁文件 → 执行恢复协议（检查output→判断状态→报告用户）
7. **反幻觉注入 (v6.0)**: 每个SubAgent prompt必须包含反幻觉5条禁令。详见 `docs/anti_hallucination_protocol.md`
8. **DM引用规范 (v6.0)**: Agent引用数据必须附带 `[DM-xxx vN.N]` 版本号。详见 `docs/data_version_control.md`

### 并行Agent Prompt规范 (v5.1+v6.0)

**每个SubAgent prompt必须包含** (v5.1):
1. **具体数据需求列表**: 列出每个数据点/指标（非泛泛描述）
2. **Phase前序发现**: 已完成Phase关键结论摘要
3. **CQ关联**: 与哪个Core Question相关
4. **字符目标**: 期望返回字符数范围
5. **输出格式要求**: 标注格式+表格+Mermaid图需求
6. **验证要求**: 标注数据来源和获取时间
7. **反幻觉5条禁令 (v6.0)**: 详见 `docs/anti_hallucination_protocol.md`
8. **DM/KAL引用指令 (v6.0)**: 详见 `docs/parallel_execution.md` v4.0
