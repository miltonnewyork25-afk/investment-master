## 3. 什么工作得不好？(改进)

### 3.1 ⚠️ VP格式不一致导致CG门控误判
- **问题**: Phase 5用VP01-VP23(无连字符), 质量门控grep pattern用`VP-[0-9]+`
- **影响**: CG5/CG6首次运行FAIL, 需debug 3轮修复
- **根因**: VP命名规范未在框架中标准化
- **修复**: grep pattern改为`VP-[0-9]+|VP[0-9]+`(alternation)
- **建议**: 框架明确规定VP命名 = `VP01`(无连字符) → 更新docs
- **优先级**: 中 — 已修复, 但需防再发

### 3.2 ⚠️ Worktree测试脚本隔离问题
- **问题**: 编辑main的`tests/quality_gate_complete.sh`不影响worktree的副本
- **影响**: 修复main脚本后worktree仍FAIL, 浪费2轮debug
- **根因**: Git worktree机制 — 未tracked文件在worktree独立存在
- **建议**:
  - 短期: 修复脚本时**同时**修改main和worktree副本
  - 长期: 考虑worktree的tests/用symlink指向main
- **优先级**: 高 — 影响所有worktree

### 3.3 ⚠️ macOS BSD grep兼容性
- **问题**: BSD grep的`-oE`不可靠支持`?`量词
- **影响**: `VP-?[0-9]+` pattern在macOS上不工作
- **建议**: 所有脚本只用`|`(alternation)替代`?`(optional)
- **优先级**: 中 — 一次性教训, 记入bash脚本规范

### 3.4 Phase 0八Agent可能过多
- **问题**: 8个专精研究Agent(地缘/技术/AI需求/估值/争议等)
- **效果**: 产出丰富但有重叠(多个Agent都搜索AI需求数据)
- **对比**: PLTR用5个Phase 0.5规划Agent产出更聚焦
- **建议**: Phase 0研究Agent ≤ 5个, 避免角度重叠
- **优先级**: 低 — 不影响质量, 只影响效率

### 3.5 Bear Case占比仍不足
- **现状**: Phase 4产出74.3K(总451K), 占比~16.5%
- **vs目标**: 18-25%目标未达(但比旧报告的10-13%有改善)
- **根因**: Phase 4是4Agent, 但其中2个是"校准"而非"看空"
- **建议**: Phase 4应有≥2个纯看空Agent(不含校准/数据核查)
- **优先级**: 高 — 系统性问题, 上次反思已提出

## 4. 发现了什么新模式？(提炼)

### 模式1: "六方法收敛" > "单一公允价值"
- **描述**: 6种独立估值方法加权收敛, 比给一个点估值更诚实更有用
- **效果**: $370收敛值 + $340-$410置信区间 = 直接可行动
- **适用范围**: 所有Tier 3报告
- **编码建议**: `docs/sotp_methodology.md` 增加"六方法收敛"标准流程

### 模式2: "CQ下调 = 对抗成功"
- **描述**: Phase 4对抗审查后CQ置信度应该下调, 下调幅度3-12pp是健康范围
- **反模式**: 如果P4后CQ全部上调 → 对抗审查太弱, 需重做
- **适用范围**: 所有Tier 3报告Phase 4
- **编码建议**: `docs/deep_dive_protocol.md` Phase 4验证条件增加"CQ下调检查"

### 模式3: "半导体OVM = 简单版"
- **描述**: 半导体公司期权少(2-3个)但概率高(>70%), 与科技平台(4-5个期权, 概率30-60%)形成对比
- **适用范围**: 半导体行业
- **编码建议**: `docs/industry/semiconductor_deep.md` 增加OVM半导体简化模板

### 模式4: "Worktree脚本隔离是系统性风险"
- **描述**: 测试脚本在worktree中有独立副本, 修改main不传播
- **适用范围**: 所有worktree操作
- **编码建议**: `docs/` 增加worktree操作注意事项, 或改用symlink
