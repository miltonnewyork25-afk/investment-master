## 3. v9.0框架验证 (LRCX + AMD共同验证) — 新章节

### 3.1 v9.0 vs v26核心差异验证

| 维度 | v26 (8报告) | v9.0 (LRCX+AMD) | 效果评估 |
|------|:---------:|:---------------:|:--------:|

| **预测跟踪** | VP(20+个) | TS追踪信号(5-8个) | "投资者应追踪X"替代"我预测X" |
| **估值表述** | 目标价点数 | 条件范围/概率加权 | LRCX $145-165(前提明确), AMD $151.6(概率加权) |
| **操作建议** | 含仓位/加减仓 | **零操作建议** | LRCX和AMD均通过零操作检查 |

| **密度(/万)** | 均值33 | 均值44.8 | **+36%**, 去掉低密度填充后自然提升 |

### 3.2 零操作建议验证
- **LRCX**: 全文零"加仓/减仓/持仓/仓位%"出现 → 通过
- **AMD**: 全文零操作触发词 → 通过
- **v9.0核心哲学**: AI优势=深度理解现在, 非预测未来。估值条件范围>点数, "好公司,错价格"替代"买入目标$X"
- **编码**: 已写入MEMORY.md和docs/

### 3.3 KS精选化 + 特异性测试
- **LRCX**: 从20+降至14个KS, 每个有明确可观测阈值
  - 通过特异性测试: "LRCX CSBG收入>$1.8B" → 替换公司名后不成立 → 特异
  - 不通过: "半导体将增长" → 替换公司名仍成立 → 删除
- **AMD**: 14个KS, Agent B执行特异性测试 → 8/12通过(67%)
  - 4个未通过的被改写为更具体版本
- **教训**: KS数量不是越多越好, 14个精选 > 20个泛泛

### 3.4 方法离散度作为不确定性指标
- **AMD**: 6方法估值范围$34.2-$151.6, 离散度4.42x → **10报告最高**
  - 含义: AMD处于AI周期+竞争格局高度不确定期
  - v9.0处理: 不给单一目标价, 报告概率加权$151.6但明确说明4.42x离散
- **LRCX**: 方法离散度<2x → 传统框架适用, 条件范围$145-165合理
- **编码**: 离散度>3x → 强烈建议不给点数目标价, 用概率加权+条件范围

### 3.5 CI注册表内嵌Phase 5 (AMD首创)
- **AMD**: 5个CI(非共识洞察)首次在Phase 5 Complete中直接嵌入
  - 此前CI注册表在Phase 3/4产出, Phase 5只引用
  - AMD Phase 5 Agent B产出CI注册表 → 直接写入Complete
- **LRCX**: CI注册表在Phase 3产出, Phase 5引用
- **建议**: CI注册表应在Phase 5中显式列出(无论产出阶段), 增强Complete自包含性
- **已编码**: CG12检测CI>=5

### 3.6 Phase 4纠错回流 (v9.0新机制)
- **机制**: Phase 4事实性纠错 → 回流覆盖Phase 1-3原文, Complete中读者只看到正确数据
- **LRCX**: 3项事实纠正回流(产能数据/市占率/客户占比)
- **AMD**: 2项事实纠正 + 7项结构性调整回流
  - Complete中Phase 4部分精简: 只保留方法论+结构性挑战, 已回流的数据纠正条目删除
- **效果**: Complete可读性大幅提升, 读者不需要"先看错误版本再看纠正"
- **编码**: `docs/deep_dive_protocol.md` Phase 4回流机制

### 3.7 Possibility Width分类器实战
- **LRCX**: 3分(窄) → 传统框架 → SOTP/DCF → 条件估值$145-165
- **AMD**: 5分(中) → **混合模式** → 传统估值(SOTP/DCF) + 可能性附录(CI注册表)
  - 首次使用"混合模式", 效果: 传统估值为主体, CI注册表补充非共识视角
  - 5分档位适合"核心业务确定但有1-2个高不确定性方向"的公司(AMD的AI GPU份额)

## 4. 什么工作得不好？(改进) — 更新版

### 4.1 Bear Case占比有改善但仍不足
- **前5报告**: 10.0%-13.4% (全部<15%)
- **TSM**: 16.5% (改善, 但仍<18%目标)
- **根因**: Phase 4的4个Agent中仅2个是纯看空, 其余是校准/数据核查
- **建议**: Phase 4 Agent中>=50%必须是纯看空(不含校准)
- **优先级**: 高 — 多次反思均提出, 改善缓慢

### 4.2 Worktree脚本隔离 (延续)
- **问题**: main的tests/脚本修改不传播到worktree副本
- **影响**: TSM浪费3轮debug才发现需要编辑worktree的脚本
- **建议**:
  - 短期: 修复脚本时同时修改main和所有worktree
  - 长期: worktree的tests/用symlink指向main
- **优先级**: 高 — 影响所有4个worktree

### 4.3 CG2检测: Phase 5 header格式问题 (新发现 — AMD痛点)
- **问题**: Phase 5 Complete的header格式需匹配CG脚本regex
- **影响**: AMD Phase 5组装后CG2未检测到Phase 5, 因为header格式不是`# Phase 5:`
- **修复**: 加了`# Phase 5:`标记后通过
- **建议**: CG脚本增加更宽松的Phase 5检测regex, 或Phase 5模板标准化header
- **优先级**: 中 — 一次性修复, 但后续报告可能再遇

### 4.4 Agent容错: P5 Agent C OAuth失败 (新发现 — AMD痛点)
- **问题**: AMD Phase 5 Agent C(CQ闭环)因OAuth token过期失败
- **影响**: 需手动补偿Agent C的产出, 增加了~30分钟额外工作
- **建议**: 关键Agent应有重试机制或手动兜底流程文档化
- **优先级**: 中 — Agent失败率仍<1%, 但关键路径上的失败代价高

### 4.5 Mermaid数量波动 (观察 — 非退步)
- **LRCX**: 124张(新高) vs **AMD**: 49张
- **分析**: 不是退步, 是内容类型差异
  - LRCX: 半导体供应链/工艺流程天然图表密集(etching/deposition流程图)
  - AMD: Phase 5 Agent产出较少图表, 且AMD竞争分析以文字对比为主
- **结论**: Mermaid数量应按内容类型设地板, 不应一刀切
  - 半导体供应链: >=60 | 竞争分析为主: >=30 | 通用: >=24(当前地板)

### 4.6 macOS BSD grep兼容性 (已解决)
- BSD grep `-oE`不可靠支持`?`量词 → 用`|`替代
- **状态**: 已编码到MEMORY.md, 不再是活跃问题

### 4.7 反思飞轮制度化 (进展中)
- **前次**: 1/8反思执行率
- **当前**: 3/10(v1.0 + v2.0 + v3.0本文件)
- **建议**: reflection.md纳入Complete组装流程(CG门控提醒)
- **优先级**: 中
